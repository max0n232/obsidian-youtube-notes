## Краткое содержание
Видео посвящено концепции Loop Engineering — новому паттерну проектирования агентских workflow, который становится актуальным в 2026 году. Автор объясняет, что простое промптинг LLM перестает быть эффективным из-за проблем масштабируемости и превращения разработчика в оператора. Loop Engineering предлагает создавать циклы, которые автоматизируют взаимодействие с агентами, проверку результатов и управление состоянием. В видео детально разбираются компоненты Loop Engineering, его риски, а также методы постепенного внедрения и примеры использования в реальных инструментах, таких как CodeSandbox.

## Ключевые моменты

*   **Проблемы текущего промптинга LLM:** В 2026 году ручное написание промтов, ожидание ответа и анализ результата плохо масштабируется. Разработчик превращается в оператора LLM, а не инженера, что ограничивает возможности текущего подхода.
*   **Концепция Loop Engineering:** Это не просто улучшенный промпт или контекст инжиниринг, а паттерн проектирования агентских workflow. Создается цикл, который автоматически запускает агента, передает задачи, проверяет результат, сохраняет состояние и решает дальнейшие действия.
*   **Истоки термина:** Термин "Loop Engineering" начал использовать Эди Османи, инженерный лидер из Google. Он отметил смещение работы программистов от промптинга к проектированию циклов. Борис Чёрный из Anthropic также высказывал схожую мысль о проектировании циклов вокруг модели.
*   **Внутренний и внешний циклы:** Внутренний цикл — это то, что агент делает по умолчанию (собирает контекст, анализирует, выполняет действия, исправляет ошибки). Внешний цикл проектирует разработчик для автоматизации запуска и управления внутренним циклом, включает триггеры, расписание, изоляцию, декомпозицию задач, управление итерациями и критерии завершения.
*   **Loop Engineering vs. обычный промптинг:** Loop Engineering отличается тем, что проектируется не отдельный запрос к модели, а система, управляющая тем, как, когда и зачем модель получает задачи. При этом качество промта и контекст инжиниринга по-прежнему важны, так как плохой промпт внутри цикла будет тиражировать ошибки.
*   **Родственная практика — Ralph-Vigop (техника Ральфа):** Это простой паттерн автономной работы агента, где он выполняет атомарные задачи, фиксирует изменения, сохраняет состояние и завершает итерацию. Следующая итерация начинается с чистого контекста. Это помогает избежать деградации продолжительных сессий, где контекстное окно переполняется. Состояние хранится вне модели (например, через `plan.md` и `status.md`).
*   **Компоненты Loop Engineering (по Эди Османи):**
    1.  **Automation (Автоматизация):** Процесс, запускающий цикл по расписанию или событию, разбирающий задачи.
    2.  **Git Worktrees:** Изолированные рабочие деревья Git для параллельной работы агентов, чтобы избежать конфликтов и хаоса.
    3.  **Skills (Навыки):** Функции для извлечения багов из логов, проверки тикетов, пополнения `todo` файла.
    4.  **Plugins & Connectors (Плагины и коннекторы):** Связывают агентов с реальными инструментами (CI/CD, таск-трекеры, базы данных) часто через MCP (Multi-Modal Orchestration Platform).
    5.  **Sub-Agents (Субагенты):** Для разделения ролей (например, паттерн Maker-Checker: один агент пишет код, другой проверяет).
    6.  **Memory (Память):** Внешний источник правды о всем процессе (Markdown файл, тикет в Jira, запись в базе данных), сохраняющий состояние между сессиями.
*   **Примеры практической реализации (CodeSandbox):**
    *   **Automations:** Запуск промтов по расписанию в изолированных Git Worktrees.
    *   **`loop` команда:** Регулярный запуск промта по расписанию (cron-подобный механизм).
    *   **Hooks и GitHub Actions:** Встраивание агента в жизненный цикл разработки для проверок и автоматизации.
    *   **`goal` команда:** Задает условия завершения работы агента. После каждого шага быстрая модель оценивает достижение условий, что требует явного встраивания тестов и проверок в цикл.
*   **Уровни автономности:**
    1.  Выполнение задач по расписанию (мониторинг багов, проверка CI).
    2.  Выполнение цели по итерациям (агент работает до достижения результата).
    3.  Независимая проверка критериев готовности (отдельный чекер или автоматическая проверка).
*   **Создание "конракта" для агента:** Включает измеримую цель (например, "довести тест coverage до 80%"), проверяемые критерии (команда `npm test` завершается с `exit code zero`), границы (что нельзя трогать), лимиты (по итерациям, времени, бюджету) и стоп-факторы.
*   **Проблема "слепоты" агента к своим ошибкам:** Агент часто считает свой результат корректным и может сгенерировать нерабочее решение. Поэтому нужен валидатор (отдельная модель, агент или тест-скрипт) для проверки (паттерн Maker-Checker).
*   **Подробнее о Git Worktrees:** Механизм для параллельной работы в одном репозитории с отдельными рабочими директориями, копиями файлов, `head` и индексом для каждого агента. Это изолирует работу на уровне файловой системы, но не отменяет конфликты слияния при интеграции.
*   **Навыки (Skills) vs. `agents.md`:** Осман предлагает определять в skills проектные правила, но автор видео считает, что для этого есть `agents.md`. В контексте Loop Engineering, skills лучше использовать для инструкций, извлекающих ошибки из логов или тикетов.
*   **Память и коннекторы:** Память хранит динамическое состояние процесса, а коннекторы через MCP дают агенту доступ к внешним системам (алертингу, мессенджерам, таск-трекерам, GitHub, CI/CD).
*   **Паттерн Maker-Checker:** Один участник (агент Maker) создает результат (например, код), а другой (агент Checker) независимо проверяет его (запускает тесты, сверяет изменения с правилами). Это снижает риск лояльности модели к своему решению. Чекерагента можно конфигурировать с конкретными условиями проверки.
*   **Матрица зрелости Loop Engineering (постепенное внедрение):**
    1.  **Уровень 1 (Ручной контроль):** Всё делается вручную, промптинг, проверка.
    2.  **Уровень 2 (Автоматизация задач):** Система разбирает задачи из трекера, формирует `todo.md`, код пишет разработчик.
    3.  **Уровень 3 (Правки агентом):** Агент вносит правки в Git Worktree, программист проверяет и мержит вручную.
    4.  **Уровень 4 (Checker):** Появляется отдельный агент-валидатор (Checker), разработчик делает финальное одобрение.
    5.  **Уровень 5 (Автомерж):** Система сама мержит изменения после прохождения проверок, разработчик контролирует по логам.
*   **Риски внедрения Loop Engineering:**
    1.  **Повышенный расход токенов и бюджета:** Плохо ограниченные циклы могут приводить к бесконечным итерациям.
    2.  **Тесты и ложное чувство безопасности:** Агент может писать тесты, подтверждающие его решение, а не реальные требования. Необходим человеческий контроль, особенно для продакшн-кода.
    3.  **Архитектурная деградация:** Агент может решать локальные задачи, но нарушать общий дизайн системы, дублировать код, создавать ненужные зависимости.
    4.  **Эффект бутылочного горлышка:** Агенты быстро генерируют код, но разработчик должен спроектировать архитектуру и атомарные задачи. Без этого агенты могут создавать "когнитивный долг". Код-ревью остается обязательным.
    5.  **Соблазн отключить критическое мышление:** Автоматизация снижает нагрузку, но не отменяет ответственность инженера.
*   **Ключевой вывод:** Loop Engineering нужно внедрять через баланс. Задачи должны быть атомарными, с измеримыми критериями. Цикл должен иметь лимиты, логи и понятный механизм передачи задачи человеку. Промты и контекст инжиниринг остаются важными, становясь частью более крупной системы.

*   **Настроить логирование и мониторинг:** Обеспечить, чтобы циклы агентов логировали свои действия и результаты, а также отправляли уведомления о стоп-факторах или проблемах для своевременного вмешательства разработчика.
